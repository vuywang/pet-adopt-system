# 四个可视化接口前后端实现说明

## 1. 文档目标

本文档说明后台首页中以下 4 个统计接口是如何从数据库取数、经过后端封装，再被前端页面渲染为图表的：

1. `GET /main/adopterGenderDistribution`
2. `GET /main/adopterAgeDistribution`
3. `GET /main/adoptDurationTrend`
4. `GET /main/petAttractionAdoptionRate`

这 4 个接口都返回统一的图表数据结构 `ChartVO`，即：

```java
public class ChartVO {
    private String name;
    private Object count;
}
```

其中：

- `name` 表示横轴标签、饼图分类名或时间标签
- `count` 表示该分类对应的统计值，可能是数量，也可能是百分比或平均值

---

## 2. 整体实现链路

这 4 个接口的整体链路是一致的，都是典型的“前端请求 -> 控制器 -> 服务层 -> Mapper -> SQL -> 返回图表数据”。

### 2.1 后端链路

后端入口在：

- `source code/api/src/main/java/com/kmbeast/controller/MainController.java`

控制器只负责暴露接口，不直接写统计逻辑。例如：

```java
@GetMapping(value = "/adopterGenderDistribution")
public Result<List<ChartVO>> adopterGenderDistribution() {
    return mainService.adopterGenderDistribution();
}
```

服务接口与实现位于：

- `source code/api/src/main/java/com/kmbeast/service/MainService.java`
- `source code/api/src/main/java/com/kmbeast/service/impl/MainServiceImpl.java`

服务层的职责很轻，主要是调用对应的 Mapper 查询，并用 `ApiResult.success(...)` 统一包装返回结果。例如：

```java
@Override
public Result<List<ChartVO>> adopterGenderDistribution() {
    return ApiResult.success(petAdoptOrderMapper.adopterGenderDistribution());
}
```

真正的统计 SQL 位于：

- `source code/api/src/main/resources/mapper/PetAdoptOrderMapper.xml`
- `source code/api/src/main/resources/mapper/PetMapper.xml`

### 2.2 前端链路

前端页面位于：

- `source code/view/src/views/admin/Main.vue`

页面在 `created()` 生命周期中主动请求这些接口：

```js
created() {
  this.fetchAdopterGenderDistribution();
  this.fetchAdopterAgeDistribution();
  this.fetchAdoptDurationTrend();
  this.fetchPetAttractionAdoptionRate();
}
```

请求完成后，前端会把后端返回的 `[{ name, count }]` 转成各图表组件可直接消费的数据：

- 饼图：转成 `[{ name, value }]`
- 柱状图：拆成 `labels` 和 `values`
- 折线图：拆成 `date` 和 `values`

使用到的图表组件有：

- `source code/view/src/components/PieCharts.vue`
- `source code/view/src/components/BarChart.vue`
- `source code/view/src/components/LineChart.vue`

## 3. 四个接口的实现分析

## 3.1 领养人性别分布接口

### 3.1.1 接口定义

后端控制器：

```java
@GetMapping(value = "/adopterGenderDistribution")
public Result<List<ChartVO>> adopterGenderDistribution() {
    return mainService.adopterGenderDistribution();
}
```

服务层：

```java
@Override
public Result<List<ChartVO>> adopterGenderDistribution() {
    return ApiResult.success(petAdoptOrderMapper.adopterGenderDistribution());
}
```

Mapper 接口：

```java
List<ChartVO> adopterGenderDistribution();
```

### 3.1.2 SQL 实现思路

SQL 位于 `PetAdoptOrderMapper.xml`。

核心逻辑：

1. 以 `pet_adopt_order` 为主表
2. 通过 `user_id` 关联 `user` 表
3. 只统计状态为 `2` 和 `4` 的订单
4. 把用户性别字段映射成图表分类名
5. 按分类聚合 `COUNT(*)`

这里的状态含义来自枚举 `PetAdoptOrderStatus`：

- `2`：审核通过
- `4`：已完成

也就是说，这个接口不是统计“所有申请人”，而是统计“已经进入成功领养流程的领养人”。

### 3.1.3 前端使用方式

前端方法：

```js
async fetchAdopterGenderDistribution() {
  const { data } = await this.$axios.get('/main/adopterGenderDistribution');
  this.adopterGenderList = data.map(entity => ({
    name: entity.name,
    value: entity.count
  }));
}
```

页面展示：

```vue
<PieCharts height="360px" :data="adopterGenderList" />
```

说明：

- 后端返回的是 `name/count`
- 饼图组件 `PieCharts` 需要的是 `name/value`
- 所以前端在请求完成后做了一次字段映射

### 3.1.4 当前数据表现

基于当前导入数据库，统计结果为：

- 女：6
- 男：5
- 未知：2

这张图适合展示“领养用户画像”的基础维度，理解门槛低，答辩时也很好讲。

## 3.2 领养人年龄段分布接口

### 3.2.1 接口定义

后端控制器：

```java
@GetMapping(value = "/adopterAgeDistribution")
public Result<List<ChartVO>> adopterAgeDistribution() {
    return mainService.adopterAgeDistribution();
}
```

服务层：

```java
@Override
public Result<List<ChartVO>> adopterAgeDistribution() {
    return ApiResult.success(petAdoptOrderMapper.adopterAgeDistribution());
}
```

Mapper 接口：

```java
List<ChartVO> adopterAgeDistribution();
```

### 3.2.2 SQL 实现思路

SQL 同样位于 `PetAdoptOrderMapper.xml`。

核心逻辑：

1. 仍然以 `pet_adopt_order` 为主表，关联 `user`
2. 仍然只统计状态为 `2`、`4` 的订单
3. 使用 `TIMESTAMPDIFF(YEAR, u.birthday, CURDATE())` 计算年龄
4. 把年龄映射到不同年龄段
5. 再对年龄段做计数
6. 使用 `sort_no` 保证年龄段展示顺序稳定

年龄段划分口径如下：

- `<= 22`
- `23 - 30`
- `31 - 40`
- `>= 41`
- `未知`

之所以额外构造 `sort_no`，是因为如果只按字符串排序，年龄段顺序可能不符合阅读习惯。这个细节对图表展示很重要。

### 3.2.3 前端使用方式

前端方法：

```js
async fetchAdopterAgeDistribution() {
  const { data } = await this.$axios.get('/main/adopterAgeDistribution');
  this.adopterAgeValues = data.map(entity => entity.count);
  this.adopterAgeLabels = data.map(entity => entity.name);
}
```

页面展示：

```vue
<BarChart
  height="360px"
  tag="领养人年龄段分布"
  :values="adopterAgeValues"
  :date="adopterAgeLabels"
/>
```

说明：

- `BarChart` 组件需要两组数组
- 一组是横轴标签 `date`
- 一组是纵轴数值 `values`
- 所以前端把后端返回的对象数组拆成了两个并行数组

### 3.2.4 当前数据表现

基于当前导入数据库，统计结果大致为：

- 23-30 岁：5
- 31-40 岁：5
- 41 岁及以上：1
- 未知：2

这张图比性别分布更能体现“用户群体画像”，也更适合和业务分析结合，例如“当前项目的主要领养用户集中在中青年群体”。

## 3.3 领养耗时趋势接口

### 3.3.1 接口定义

后端控制器：

```java
@GetMapping(value = "/adoptDurationTrend")
public Result<List<ChartVO>> adoptDurationTrend() {
    return mainService.adoptDurationTrend();
}
```

服务层：

```java
@Override
public Result<List<ChartVO>> adoptDurationTrend() {
    return ApiResult.success(petAdoptOrderMapper.adoptDurationTrend());
}
```

Mapper 接口：

```java
List<ChartVO> adoptDurationTrend();
```

### 3.3.2 业务口径说明

这个接口中的“领养耗时”不是指审核时长，也不是指用户提交申请到审核结束的时间。

它统计的是：

**宠物发布时间 `pet.create_time` 到领养订单创建时间 `pet_adopt_order.create_time` 之间的天数差。**

因此它更准确的业务含义是：

**一只宠物从进入待领养状态，到被用户领养申请/完成之间，大约经历了多久。**

### 3.3.3 SQL 实现思路

核心逻辑：

1. 关联 `pet_adopt_order` 与 `pet`
2. 只统计状态为 `2`、`4` 的订单
3. 要求宠物发布时间不为空
4. 要求订单创建时间晚于等于宠物发布时间，避免出现负值
5. 用 `DATEDIFF(pao.create_time, p.create_time)` 计算每条记录的耗时天数
6. 按月份 `DATE_FORMAT(..., '%Y-%m')` 分组
7. 对每月耗时取平均值

最终返回的数据是：

- `name`：月份，例如 `2025-06`
- `count`：该月平均领养耗时，单位为天

### 3.3.4 前端使用方式

前端方法：

```js
async fetchAdoptDurationTrend() {
  const { data } = await this.$axios.get('/main/adoptDurationTrend');
  this.adoptDurationValues = data.map(entity => entity.count);
  this.adoptDurationDates = data.map(entity => entity.name);
}
```

页面展示：

```vue
<LineChart
  height="360px"
  tag="平均领养耗时趋势"
  subtext="平均耗时(天)"
  :values="adoptDurationValues"
  :date="adoptDurationDates"
  :show-controls="false"
/>
```

说明：

- 这里使用的是折线图 `LineChart`
- `show-controls="false"` 表示这张图没有时间范围切换控件
- 因为后端 SQL 已经直接按月聚合，不再需要像订单趋势图那样由前端选择最近 30/90/365 天

### 3.3.5 当前数据表现

当前数据库中能统计出的样本月份包括：

- 2025-06：9.0 天
- 2025-07：12.5 天
- 2025-08：26.0 天
- 2025-09：24.0 天
- 2025-10：12.0 天
- 2025-11：22.0 天
- 2025-12：12.0 天
- 2026-02：14.0 天
- 2026-03：23.0 天

这张图特别适合用来讲“平台运营效率”或“宠物被领养速度”的变化。

## 3.4 宠物吸引力与领养率接口

### 3.4.1 接口定义

后端控制器：

```java
@GetMapping(value = "/petAttractionAdoptionRate")
public Result<List<ChartVO>> petAttractionAdoptionRate() {
    return mainService.petAttractionAdoptionRate();
}
```

服务层：

```java
@Override
public Result<List<ChartVO>> petAttractionAdoptionRate() {
    return ApiResult.success(petMapper.petAttractionAdoptionRate());
}
```

Mapper 接口：

```java
List<ChartVO> petAttractionAdoptionRate();
```

### 3.4.2 业务口径说明

这个接口本质上是在回答一个问题：

**互动热度更高的宠物，是否更容易被领养？**

它并没有直接识别“颜值”，而是用用户行为数据构造了一个“吸引力分数”。

所以如果在汇报时使用“宠物颜值影响被领养图”这个说法，建议更严谨地表述为：

**宠物吸引力/互动热度与领养率关系图**

这样和实际实现更加一致。

### 3.4.3 SQL 实现思路

SQL 位于 `PetMapper.xml`，是 4 个接口中业务含义最强的一个。

核心分成两层：

#### 第一层：计算每只宠物的吸引力得分

从 `active_net` 表中提取 `content_type = 'PET'` 的互动行为，按 `content_id` 聚合：

- 浏览 `type = 1` 记 1 分
- 点赞 `type = 2` 记 2 分
- 收藏 `type = 3` 记 3 分

得到每只宠物的总分 `score`。

#### 第二层：把宠物分桶并计算各桶领养率

然后把宠物按照得分分成四个区间：

- `0-2`
- `3-5`
- `6-9`
- `10+`

同时取 `pet.is_adopt` 作为是否已领养标记：

- `is_adopt = 1` 表示已领养
- `is_adopt = 0` 表示未领养

最后按分桶聚合，计算：

```sql
SUM(is_adopt) / COUNT(*) * 100
```

这就是每个吸引力区间的领养率。

### 3.4.4 前端使用方式

前端方法：

```js
async fetchPetAttractionAdoptionRate() {
  const { data } = await this.$axios.get('/main/petAttractionAdoptionRate');
  this.petAttractionValues = data.map(entity => entity.count);
  this.petAttractionLabels = data.map(entity => entity.name);
}
```

页面展示：

```vue
<BarChart
  height="360px"
  tag="宠物吸引力与领养率"
  :values="petAttractionValues"
  :date="petAttractionLabels"
/>
```

说明：

- 横轴是吸引力分桶
- 纵轴是对应的领养率
- 使用柱状图比折线图更适合展示离散分组对比

### 3.4.5 当前数据表现

根据当前数据库样本，四个分桶的结果为：

- `0-2`：领养率 `0.0%`，样本数 20
- `3-5`：领养率 `0.0%`，样本数 6
- `6-9`：领养率 `40.0%`，样本数 5
- `10+`：领养率 `66.7%`，样本数 9

从当前样本可以得到一个非常直观的展示结论：

**互动热度越高的宠物，领养率越高。**

这张图是 4 个接口里最有“业务故事感”的一张，也最适合用于答辩展示。

## 4. 前端图表渲染方式总结

这 4 个接口虽然都返回 `ChartVO`，但前端会根据图表类型做不同处理。

### 4.1 饼图类

适用于：

- `adopterGenderDistribution`

前端需要把后端字段从 `count` 改成 `value`，因为 `PieCharts.vue` 组件约定的是：

```js
[{ name: '女', value: 6 }]
```

### 4.2 柱状图类

适用于：

- `adopterAgeDistribution`
- `petAttractionAdoptionRate`

前端把对象数组拆成：

- 标签数组
- 数值数组

对应 `BarChart.vue` 的 `date` 与 `values` 属性。

### 4.3 折线图类

适用于：

- `adoptDurationTrend`

同样拆成：

- 时间数组
- 数值数组

对应 `LineChart.vue` 的 `date` 与 `values` 属性。

---

## 5. 这 4 个接口为什么适合做后台首页图表

从实现成本、数据可得性和展示效果来看，这 4 张图搭配得比较合理：

1. `adopterGenderDistribution`
   适合做基础用户画像，结果容易理解。

2. `adopterAgeDistribution`
   比性别更有分析价值，可以体现领养主力年龄段。

3. `adoptDurationTrend`
   体现平台运营结果和宠物被领养速度，适合做趋势分析。

4. `petAttractionAdoptionRate`
   把行为数据和业务结果关联起来，最能体现“数据驱动”的感觉。

它们组合起来，刚好覆盖了：

- 用户画像
- 时间趋势
- 业务效率
- 行为与结果关系

所以不只是“能画图”，而是形成了一套比较完整的统计视角。

## 6. 当前实现的优点

### 6.1 后端职责清晰

控制器、服务层、Mapper 的分工很明确：

- Controller 只暴露接口
- Service 只做调用和包装
- Mapper XML 承担具体统计 SQL

这种方式对课程项目和中小型后台系统都很合适。

### 6.2 前端接入成本低

后端统一返回 `ChartVO`，前端只需做很轻量的数据转换，就能复用已有的饼图、柱状图、折线图组件。

### 6.3 展示效果与业务含义兼顾

这 4 个接口并不是纯技术性的统计，而是都能对应到业务问题：

- 谁在领养
- 哪些年龄段更活跃
- 宠物多久被领养
- 互动热度是否影响领养结果

---

## 7. 注意事项与可优化点

### 7.1 “颜值影响”目前是推断，不是直接字段

当前系统没有宠物颜值评分字段，也没有图像模型打分。

因此 `petAttractionAdoptionRate` 实际反映的是：

- 浏览量
- 点赞量
- 收藏量

共同构成的“互动吸引力”

如果汇报时直接说成“颜值影响”，容易被老师追问实现依据。更稳妥的说法是：

**互动吸引力与领养率关系图**

### 7.2 领养耗时口径要提前讲清楚

当前耗时的计算口径是：

- 宠物发布时间
- 到领养订单创建时间

并不是：

- 审核耗时
- 用户提交到完成的流程耗时

建议在演示或文档中把这个定义提前说明。

### 7.3 当前样本量适合展示，不适合做严谨统计结论

当前数据库中：

- 领养订单总数为 13
- 成功样本主要集中在少量月份

因此这些图更适合用于：

- 项目演示
- 功能展示
- 业务分析思路说明

不适合上升为严格的真实业务结论。

### 7.4 中文展示文本要留意编码问题

当前部分源码文件在控制台查看时出现乱码痕迹，但从结构上看逻辑是清楚的。若后续页面出现中文显示异常，需要统一检查：

- 源文件编码
- 数据库字符集
- IDE 保存编码

## 8. 结论

这 4 个接口已经形成了一条完整的可视化实现链：

1. 后端通过 `MainController` 暴露统计接口
2. `MainServiceImpl` 负责调度查询并统一返回 `Result<List<ChartVO>>`
3. `PetAdoptOrderMapper.xml` 与 `PetMapper.xml` 编写统计 SQL
4. 前端 `Main.vue` 在页面创建时主动请求接口
5. 前端把通用 `ChartVO` 数据转换成图表组件所需格式
6. 最终由 `PieCharts`、`BarChart`、`LineChart` 渲染成首页可视化卡片

从课程项目视角看，这套方案的优点是：

- 实现难度适中
- 业务解释性强
- 页面展示效果好
- 很适合答辩时讲“我不仅做了 CRUD，还做了统计分析和可视化”

如果后续继续扩展，优先建议在这套体系上增加：

1. 按宠物类别的领养率对比
2. 宠物发布地分布或领养地分布
3. 宠物互动量月趋势
4. 宠物帖子互动量与宠物领养转化的关联分析
