# 推荐算法改造交付说明

## 1. 文档说明

本文用于说明本次推荐模块改造的目标、改造前后算法差异、涉及的系统模块及验证结果。

说明范围如下：

- 说明原有推荐算法的业务思路
- 说明当前替换后的推荐算法思路
- 说明改造影响到的模块和文件范围
- 说明本次改造后的验证情况

本文不包含具体实现细节、特征构建规则、具体计算过程和可直接复刻的实现代码。

## 2. 改造背景

原项目的推荐模块采用的是基于用户的协同过滤思路，即根据用户之间的行为相似度，推断当前用户可能感兴趣的宠物内容或宠物经验帖子。

本次改造需求明确如下：

- 放弃使用 `UserCF`
- 改为使用 `基于内容的推荐算法`
- 保留 `热门内容兜底` 机制
- 不额外引入中间件，基于现有 Spring Boot + MyBatis + MySQL 架构完成改造

## 3. 基于内容的推荐算法说明

### 3.1 算法类型

本次替换后的推荐算法属于：

- 基于内容的推荐算法

在表达上，可概括为：

- `TF-IDF + 余弦相似度`

更准确地说，本次算法思想是：

- 对宠物或帖子内容做特征向量化
- 将用户历史偏好汇总为用户兴趣向量
- 通过余弦相似度衡量候选内容与用户兴趣的接近程度
- 根据相似度高低输出推荐结果

### 3.2 为什么选择该算法

相较于协同过滤，基于内容的推荐更适合当前项目，主要原因如下：

- 当前项目中的内容对象具有较明确的结构化属性
- 宠物和帖子本身包含较丰富的文本描述信息
- 宠物领养行为整体偏低频，不适合过度依赖用户之间的相似关系
- 对新用户或低活跃用户场景，内容推荐更容易给出稳定结果

### 3.3 TF-IDF + 余弦相似度的基本原理

该算法可以理解为两个阶段：

#### 第一阶段：内容向量化

系统会先把推荐对象转化成一组可计算的特征表示。

在文本场景中，常见做法是：

- 提取标题、摘要、描述、正文中的关键词
- 通过 TF-IDF 的方式衡量某个词在当前内容中的重要程度
- 将一条内容表示为一个特征向量

其中：

- `TF` 表示词频，反映某个词在当前内容中出现得是否足够多
- `IDF` 表示逆文档频率，反映某个词在全体内容中是否具有区分度
- `TF-IDF` 的值越高，说明这个词越能代表这条内容的特征

在本项目的业务场景中，除了文本信息外，还会结合结构化内容特征共同参与画像和匹配，例如：

- 宠物类别
- 年龄段
- 性别
- 地区
- 是否接种疫苗
- 帖子所属宠物类别

因此，本项目中的“内容向量化”并不只依赖纯文本词频，还包含业务属性特征。

#### 第二阶段：相似度计算

当内容被表示为向量后，系统再将用户历史偏好内容进行汇总，形成一个用户兴趣向量。

随后，针对每一个候选内容，计算它与用户兴趣向量之间的余弦相似度。

余弦相似度的含义是：

- 不直接比较两个向量的绝对大小
- 而是比较两个向量方向是否接近
- 越接近，说明候选内容与用户偏好越相似

因此：

- 相似度高的内容会优先推荐
- 相似度低的内容会被降到后面

### 3.4 在本项目中的使用方式

本项目中，`TF-IDF + 余弦相似度` 的使用方式可以概括为以下思路：

1. 从宠物信息或帖子信息中提取内容特征
2. 从用户历史浏览、点赞、收藏行为中识别其偏好内容
3. 结合用户交互过的内容，汇总用户兴趣方向
4. 将候选内容与用户兴趣做相似度匹配
5. 按相似度高低输出推荐结果
6. 当个性化推荐不足时，回退到热门内容

也就是说，本项目中的行为表仍然保留使用，但其作用不再是构建“相似用户关系”，而是：

- 识别用户喜欢过什么内容
- 提炼用户偏好的内容特征
- 为内容推荐提供用户兴趣来源

## 3. 原有算法说明

### 3.1 原算法类型

原有算法属于：

- 个性化推荐算法
- 协同过滤算法
- 其中具体为 `基于用户的协同过滤（User-Based Collaborative Filtering）`

### 3.2 原算法业务思路

原系统通过用户在行为表中的浏览、点赞、收藏等记录，构建用户对内容的兴趣分值，然后基于用户与用户之间的相似关系，生成推荐结果。

原有逻辑主要体现为：

- 从行为数据中抽取用户与内容之间的交互关系
- 对交互行为做加权处理
- 构建用户与内容之间的兴趣矩阵
- 计算用户之间的相似度
- 将相似用户感兴趣的内容推荐给当前用户

### 3.3 原算法适用特点

原算法的优点：

- 能体现一定的个性化推荐能力
- 对已有行为数据的利用较直接
- 在用户行为量较大时，推荐结果容易体现“相似用户也喜欢”的效果

原算法的不足：

- 对用户行为数据依赖较强
- 新用户或低活跃用户场景下效果不稳定
- 容易受到数据稀疏影响
- 在宠物领养这种低频行为场景下，用户相似关系未必稳定

## 4. 当前算法说明

### 4.1 当前算法类型

本次替换后的算法为：

- `基于内容的推荐算法（TF-IDF + 余弦相似度）`

整体业务思路为：

- 根据用户历史行为对应的内容特征，构建用户兴趣画像
- 将候选内容的内容特征与用户兴趣画像进行相似度匹配
- 输出与当前用户兴趣更接近的宠物或帖子
- 当个性化结果不足时，回退到热门内容推荐

### 4.2 当前算法的适用性

相对于原有 UserCF，本次采用内容推荐更适合当前项目，原因包括：

- 宠物领养属于低频行为场景，用户间稳定相似关系较弱
- 系统已有宠物类型、年龄、性别、地区、疫苗状态、标题、摘要、描述等内容特征，适合做内容匹配
- 内容推荐对数据稀疏问题更友好
- 在当前项目体量下更容易维护和解释

### 4.3 当前算法在本项目中的落地思路

为便于项目说明，可以将当前算法的落地过程概括为以下四步：

#### 1. 内容特征抽取

系统先从宠物和帖子中提取可表达内容语义的特征。

宠物内容主要关注：

- 宠物类别
- 年龄
- 性别
- 所在地区
- 疫苗状态
- 名称与描述信息

帖子内容主要关注：

- 标题
- 摘要
- 正文内容
- 所属宠物类别

#### 2. 用户兴趣画像构建

系统会根据用户历史行为记录，识别其曾经关注过哪些宠物或帖子。

这些内容的特征会被汇总为用户兴趣画像。

因此，用户的浏览、点赞、收藏行为在当前算法下的作用是：

- 不再直接用于找相似用户
- 而是用于提炼用户个人兴趣方向

#### 3. 候选内容相似度计算

对每条候选内容，系统都会将其内容向量与用户兴趣画像向量进行比对。

在比对阶段使用余弦相似度，主要目的在于衡量：

- 这条内容与用户历史偏好是否足够接近
- 它与用户当前兴趣方向的一致程度

#### 4. 推荐结果输出与兜底

当候选内容经过相似度排序后，系统输出靠前结果作为个性化推荐结果。

若当前用户历史行为不足，或者内容匹配结果不稳定，则继续回退到热门内容，保证推荐接口始终可以返回有效数据。

### 4.3 当前算法与热门兜底的关系

本次没有取消原系统中的热门兜底逻辑。

当前推荐链路分为两层：

1. 优先执行基于内容的个性化推荐
2. 当个性化推荐结果为空或不足时，回退到热门内容结果

这样可以保证：

- 有历史行为的用户优先得到个性化结果
- 历史行为不足时，系统仍能稳定返回可展示内容

## 5. 原算法与当前算法的区别

### 5.1 核心差异

| 对比项 | 原算法 | 当前算法 |
| --- | --- | --- |
| 算法类型 | 基于用户的协同过滤 | 基于内容的推荐 |
| 推荐依据 | 相似用户的历史行为 | 内容特征与用户兴趣画像 |
| 对用户行为依赖 | 高 | 中等 |
| 对内容特征依赖 | 低 | 高 |
| 冷启动表现 | 较弱 | 相对更稳 |
| 业务解释性 | 一般 | 较强 |
| 与宠物领养场景适配度 | 一般 | 更高 |

### 5.2 数据使用方式差异

原算法中，行为表主要用于计算“人和人之间”的关系。

当前算法中，行为表的作用发生了变化：

- 不再用于计算用户之间的相似度
- 转而用于提取当前用户偏好的内容范围
- 用户行为变成“画像来源”，而不是“相似用户匹配来源”

### 5.3 兜底机制差异

原系统已存在热门内容回退逻辑，本次改造保留该逻辑，没有取消。

因此本次改造不是完全重做推荐链路，而是将：

- 原来的 `UserCF 主推荐`

替换为：

- `基于内容的推荐（TF-IDF + 余弦相似度）主推荐`

同时继续保留：

- `热门内容兜底`
