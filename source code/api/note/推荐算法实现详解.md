# 推荐算法实现详解

## 1. 方案定位

当前项目的“推荐”不是传统协同过滤主链路，而是以“内容画像 + 用户行为加权 + 余弦相似度排序”为核心的内容推荐。

代码里实际上存在两条不同的推荐展示逻辑：

1. **手动推荐位**：后台在宠物数据上打 `is_recommend` 标记，前端首页直接展示这一批内容。
2. **智能推荐位**：基于 `active_net` 里的用户行为，拼出用户兴趣画像，再对候选内容做相似度排序。

宠物和宠物经验帖两套业务都复用了同一套推荐思路，只是在特征字段和兜底规则上略有差异。

---

## 2. 数据来源

### 2.1 内容数据

推荐系统的内容源来自两张主表：

1. `pet`
   - 对应实体：`com.kmbeast.pojo.entity.Pet`
   - 推荐候选读取：`PetMapper.queryRecommendCandidates()`
   - 关键字段：`id、name、detail、address、age、gender、pet_type_id、is_vaccine、is_adopt`

2. `pet_post`
   - 对应实体：`com.kmbeast.pojo.entity.PetPost`
   - 推荐候选读取：`PetPostMapper.queryRecommendCandidates()`
   - 关键字段：`id、pet_type_id、title、summary、content、is_audit`

### 2.2 行为数据

用户行为统一落在 `active_net` 表：

- 对应实体：`com.kmbeast.pojo.entity.ActiveNet`
- MyBatis Mapper：`com.kmbeast.mapper.ActiveNetMapper`
- 关键字段：`user_id、content_id、content_type、type、create_time`

行为类型在代码注释里已经约定为：

- `type = 1`：浏览
- `type = 2`：点赞
- `type = 3`：收藏

项目中会在详情页打开时写入浏览记录：

- `PetServiceImpl.getById()` 会在用户首次浏览宠物详情时插入 `active_net`
- `PetPostServiceImpl.getById()` 会在用户首次浏览经验帖详情时插入 `active_net`

这两个方法都先通过 `ActiveNetMapper.queryCount(...)` 判断是否已经存在同一用户、同一内容、同一行为类型的记录，避免重复写入浏览行为。

### 2.3 手动推荐标记

宠物实体上还有一个 `is_recommend` 字段：

- `PetServiceImpl.recommend()` 直接按 `is_recommend = true` 查询
- `PetMapper.queryListItem(...)` 通过 `PetQueryDto.isRecommend` 过滤

这部分是“人工配置推荐位”，不是行为驱动的智能推荐。

## 3. 用户行为打分

### 3.1 行为权重映射

`ContentBasedRecommendUtil.behaviorWeight(Integer type)` 把行为转成权重：

- 浏览 `1 -> 1.0`
- 点赞 `2 -> 2.0`
- 收藏 `3 -> 3.0`

这意味着收藏对画像的影响最大，点赞次之，浏览最弱。

### 3.2 行为聚合方式

在 `PetServiceImpl.aggregateInteractionWeights(...)` 和 `PetPostServiceImpl.aggregateInteractionWeights(...)` 里，会遍历用户历史行为：

1. 先用 `ContentBasedRecommendUtil.behaviorWeight(type)` 得到每条行为的权重
2. 再按 `content_id` 做累加

所以同一内容如果被多次互动，权重会叠加，不是简单去重。

### 3.3 `queryScore` 的含义

`ActiveNetMapper.queryScore(String contentType)` 的 SQL 会把行为分数汇总成：

- 浏览 `1`
- 点赞 `1`
- 收藏 `3`

按 `user_id + content_id` 分组后输出 `ScoreVO(userId, contentId, score)`。

但就当前代码来说，`queryScore` 没有被推荐主流程调用，更多是一个可复用的统计口径或后续扩展入口。当前智能推荐的实际实现用的是 `aggregateInteractionWeights(...)` + `behaviorWeight(...)` 这套内存内聚合逻辑。

---

## 4. 候选集构建

### 4.1 宠物候选集

宠物推荐候选来自 `PetMapper.queryRecommendCandidates()`，SQL 直接从 `pet` + `pet_type` 取字段，构造 `PetRecommendCandidateVO`。

在 `PetServiceImpl.buildPetContentRecommendations(...)` 中还会再过滤一次：

- `!Boolean.TRUE.equals(item.getIsAdopt())`

也就是说，已经领养的宠物不会进入候选集。

### 4.2 经验帖候选集

经验帖推荐候选来自 `PetPostMapper.queryRecommendCandidates()`，构造 `PetPostRecommendCandidateVO`。

在 `PetPostServiceImpl.buildPetPostContentRecommendations(...)` 中会过滤：

- `Boolean.TRUE.equals(item.getIsAudit())`

也就是说，只有已审核帖子才会进入推荐候选集。

### 4.3 候选集的“去重排除”

推荐时会把用户已经互动过的内容排除掉：

- 宠物：`excludeIds = new HashSet<>(interactionWeights.keySet())`
- 经验帖：同样使用 `interactionWeights.keySet()`

随后 `ContentBasedRecommendUtil.rankItems(...)` 会直接过滤这些 ID，避免重复推荐用户已经浏览/点赞/收藏过的内容。

---

## 5. 特征表示与相似度排序

### 5.1 宠物特征向量

`PetServiceImpl.buildPetFeatureVector(PetRecommendCandidateVO candidate)` 把一条宠物记录转成特征向量，主要包括：

- 类别特征
  - `petTypeId`，权重 `3`
  - `petTypeName`，权重 `2`
  - `gender`，权重 `1.5`
  - `isVaccine`，权重 `1.5`
- 数值分桶特征
  - `age` 通过 `ContentBasedRecommendUtil.addBucketFeature(...)`
  - 规则：`<=6`、`<=24`、`<=72`、`>72` 对应 `baby / young / adult / senior`
- 文本特征
  - `name`，权重 `2`
  - `address`，权重 `1.5`
  - `detail`，权重 `1`

### 5.2 经验帖特征向量

`PetPostServiceImpl.buildPetPostFeatureVector(PetPostRecommendCandidateVO candidate)` 使用的字段更少：

- 类别特征
  - `petTypeId`，权重 `3`
  - `petTypeName`，权重 `2`
- 文本特征
  - `title`，权重 `2.5`
  - `summary`，权重 `2`
  - `content`，权重 `1`

### 5.3 文本切分规则

`ContentBasedRecommendUtil.addTextFeatures(...)` 先把 HTML 转纯文本，再分词：

- 英文和数字：按连续字符串切词，长度至少 2 才保留
- 中文：同时保留单字和双字特征

这对中文短文本很实用，能在没有外部分词器时保住一部分语义信号。

### 5.4 用户画像构建

`ContentBasedRecommendUtil.buildUserProfile(...)` 的逻辑很直接：

1. 取用户历史里每个内容的特征向量
2. 用该内容的行为权重做乘法
3. 累加到用户画像上

所以用户画像本质上是“历史高权重内容特征的加权和”。

### 5.5 排序方式

`ContentBasedRecommendUtil.rankItems(...)` 的排序步骤是：

1. 过滤掉用户已交互内容
2. 计算候选内容向量与用户画像的余弦相似度
3. 只保留相似度 `> 0` 的内容
4. 按相似度降序排序
5. 只取前 `count` 条

这就是当前智能推荐的核心。

## 6. 冷启动与兜底

### 6.1 用户没有历史行为

如果用户没有 `active_net` 历史，`buildPetContentRecommendations(...)` / `buildPetPostContentRecommendations(...)` 会直接返回空列表。

### 6.2 用户画像为空

`ContentBasedRecommendUtil.rankItems(...)` 里如果 `userProfile.isEmpty()`，也会返回空列表。

这意味着当前算法没有“纯新用户画像”机制。

### 6.3 热门兜底

当智能推荐结果为空时，`PetServiceImpl.autoRecommend(...)` 和 `PetPostServiceImpl.autoRecommend(...)` 都会走兜底逻辑：

1. 调 `ActiveNetMapper.queryAllIds(contentType, ActiveNetType.VIEW.getStatus(), count)`
2. 按浏览量最多取一批热门内容
3. 再用 `queryListItemByIds(...)` 查详细列表
4. 对宠物执行 `filterAvailablePets(...)`
5. 对经验帖执行 `filterAuditedPosts(...)`

这里的兜底不是“最相似”，而是“最热门浏览”，属于曝光型回退方案。

### 6.4 首页 Banner 的兜底

`PetServiceImpl.recommend()` 还有一条更简单的首页推荐逻辑：

1. 优先查询 `is_recommend = true` 的宠物
2. 如果没有推荐位内容，就取前 3 条普通宠物作为默认展示

这条链路不是行为推荐，而是首页展示位的静态回退。

## 7. 前后端调用链

### 7.1 首页宠物 Banner

前端：`source code/view/src/views/view/Index.vue`

- `created()` 里调用 `fetchRecommendPetInfo()`
- 请求：`GET /pet/recommend`

后端链路：

`PetController.recommend()` -> `PetServiceImpl.recommend()` -> `PetMapper.queryListItem(...)`

### 7.2 用户端宠物智能推荐

前端：`source code/view/src/views/user/PetList.vue`

- `created()` 里调用 `fetchAutoRecommend()`
- 请求：`GET /pet/autoRecommend/4`

后端链路：

`PetController.autoRecommend(count)` -> `PetServiceImpl.autoRecommend(count)` -> `buildPetContentRecommendations(...)` -> `PetMapper.queryRecommendCandidates()` / `ActiveNetMapper.query(...)` / `ContentBasedRecommendUtil.rankItems(...)`

### 7.3 用户端经验帖智能推荐

前端：`source code/view/src/views/user/PetPost.vue`

- `created()` 里调用 `fetchRecommendPetPostList()`
- 请求：`GET /pet-post/autoRecommend/3`

后端链路：

`PetPostController.autoRecommend(count)` -> `PetPostServiceImpl.autoRecommend(count)` -> `buildPetPostContentRecommendations(...)` -> `PetPostMapper.queryRecommendCandidates()` / `ActiveNetMapper.query(...)` / `ContentBasedRecommendUtil.rankItems(...)`

### 7.4 交互行为采集

推荐依赖的行为数据主要来自详情页“浏览”采集：

- `PetServiceImpl.getById()`
- `PetPostServiceImpl.getById()`

两者都会在首次浏览时向 `active_net` 写入记录，为后续推荐提供训练/画像依据。

## 9. 局限性

当前实现能跑通“内容推荐闭环”，但仍然有明显边界：

1. **不是协同过滤主链路**
   - 项目里有 `UserBasedCFUtil`，但没有接入实际推荐页面
   - 当前真正生效的是内容推荐

2. **没有时间衰减**
   - 过去很久的浏览、点赞、收藏和最近行为同权处理
   - 用户兴趣变化无法自动降权

3. **没有负反馈**
   - 没有“不感兴趣/屏蔽/跳过”一类信号
   - 画像只能越来越“正向强化”

4. **排序只看相似度**
   - 没有覆盖率、多样性、去重复曝光等策略
   - 结果容易集中在同一类宠物或帖子上

5. **兜底偏热门**
   - 热门回退依赖浏览量，可能强化头部内容
   - 冷启动下对长尾内容不友好

6. **候选集直接扫全表**
   - `queryRecommendCandidates()` 不做预筛
   - 数据量上来后，内存特征计算和余弦排序会有性能压力

7. **`queryScore` 目前未接入主流程**
   - 有统计能力，但没有直接参与推荐排序
   - 如果后续要做“显式评分推荐”，它可以作为扩展入口

---

## 10. 答辩时可以怎么概括

一句话概括就是：

> 本项目把用户在 `active_net` 中的浏览、点赞、收藏行为转成权重，结合宠物/帖子内容特征构建用户画像，再用余弦相似度对候选内容排序；当画像不足时，退回到热门浏览内容，保证推荐可用。

如果需要展开，可以按这四步讲：

1. 先说明数据怎么来
2. 再说明行为怎么打分
3. 再说明候选怎么建
4. 最后说明推荐失败时怎么兜底
