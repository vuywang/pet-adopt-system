# 答辩展示：可视化与推荐算法说明

## 1. 文档目的

本文用于说明本次宠物领养系统为答辩展示所新增的可视化内容、统计口径与推荐算法流程，重点解决以下两个问题：

1. 系统不仅能够完成基础业务流程，还能够对平台数据进行分析展示。
2. 系统中的推荐功能不是黑盒调用，而是有明确的输入、处理过程和输出逻辑。

正文以业务说明为主，不直接展开源码实现。核心代码位置统一放在附录中，便于老师查阅。

## 2. 本次新增内容概述

本次改动主要分为三部分：

1. 新增领养数据分析图表，补充后台首页的展示维度。
2. 增强答辩演示数据，保证图表与趋势分析具有可读性。
3. 形成一份面向答辩的算法说明文档，清楚说明推荐逻辑。

## 3. 可视化图表设计说明

### 3.1 领养订单创建趋势

该图用于展示平台在一段时间内领养订单的变化情况，反映平台活跃度与领养申请波动趋势。

统计口径：

1. 数据来源为 `pet_adopt_order` 表。
2. 按日期统计指定时间范围内的订单创建数量。
3. 后台支持按最近 30 天、90 天、365 天切换查看。

答辩时可以说明：

1. 平台不是静态展示页，而是具备持续产生业务数据的能力。
2. 订单趋势图适合用于观察活动投放、热门宠物上线后的业务变化。

### 3.2 宠物类别分布

该图用于说明当前平台宠物资源结构，展示猫、狗等不同类别的数量分布。

统计口径：

1. 数据来源为 `pet` 与 `pet_type` 表。
2. 以宠物类别为维度进行汇总统计。

答辩时可以说明：

1. 平台资源不是单一类型，具备多类别宠物管理能力。
2. 图表也可以为后续运营推荐、资源配置提供依据。

### 3.3 领养人性别分布

该图用于展示有效领养订单对应用户的性别结构，体现平台用户画像分析能力。

统计口径：

1. 数据来源为 `pet_adopt_order` 与 `user` 表。
2. 统计状态为“已审核”或“已完成”的有效订单。
3. 用户性别分为男性、女性、未知三类。

答辩时可以说明：

1. 系统已经具备对领养用户做基础画像分析的能力。
2. 这类数据有助于后续平台内容运营和宠物信息展示优化。

### 3.4 领养人年龄段分布

该图用于展示领养用户的年龄段集中情况，比单纯展示用户列表更能体现数据分析能力。

统计口径：

1. 数据来源为 `pet_adopt_order` 与 `user` 表。
2. 通过用户生日计算年龄，并划分为 `22岁及以下`、`23-30岁`、`31-40岁`、`41岁及以上`、`未知` 五档。
3. 只统计有效领养订单对应的用户。

答辩时可以说明：

1. 当前系统可以从业务数据进一步抽取用户结构特征。
2. 平台后续可以针对不同年龄层推荐不同类型的宠物内容或领养指南。

### 3.5 平均领养耗时趋势

该图用于展示宠物从发布到进入有效领养订单的平均耗时变化，属于老师提到的“曲线类分析图”。

统计口径：

1. 数据来源为 `pet` 与 `pet_adopt_order` 表。
2. 以订单创建时间所在月份为横轴。
3. 计算 `订单创建时间 - 宠物发布时间` 的平均天数，作为平均领养耗时。

答辩时可以说明：

1. 系统不仅展示数量，还可以分析领养效率。
2. 如果后续平台优化详情页、推荐页或活动投放，可以通过该图观察平均领养周期是否缩短。

### 3.6 宠物吸引力与领养率

老师原本提出“宠物颜值影响被领养图”，但系统数据库中没有直接的“颜值评分”字段。因此本系统采用更合理、更可落地的代理指标：`宠物吸引力指数`。

吸引力指数定义如下：

1. 浏览记为 1 分。
2. 点赞记为 2 分。
3. 收藏记为 3 分。

统计口径：

1. 数据来源为 `pet` 与 `active_net` 表。
2. 先计算每只宠物的互动加权得分。
3. 再将宠物划分为 `低吸引力(0-2)`、`中等吸引力(3-5)`、`较高吸引力(6-9)`、`高吸引力(10+)` 四档。
4. 统计各档宠物的被领养率。

答辩时可以说明：

1. 虽然系统没有直接存储颜值分数，但可以利用平台真实互动行为构造更贴近业务的吸引力指标。
2. 这张图能够体现“高互动宠物更容易被领养”的关系，更适合作为数据分析结论。

## 4. 推荐算法过程说明

### 4.1 算法定位

当前系统主推荐逻辑采用的是：

1. 基于内容的推荐算法。
2. 行为加权的用户兴趣画像。
3. 余弦相似度排序。
4. 热门内容兜底返回。

这意味着系统不是直接寻找“和你相似的其他用户”，而是优先分析“你过去对什么内容感兴趣”，再寻找与这些内容特征更接近的新宠物或帖子。

### 4.2 推荐输入

系统推荐所依赖的输入主要有两类：

1. 用户行为数据。
2. 内容特征数据。

用户行为数据来自 `active_net` 表，当前记录三类行为：

1. 浏览
2. 点赞
3. 收藏

内容特征数据来自宠物和帖子本身：

1. 宠物特征：类别、类别名称、性别、年龄段、是否接种疫苗、名称、地区、详情描述。
2. 帖子特征：宠物类别、标题、摘要、正文内容。

### 4.3 用户行为加权

系统不会把三类行为看成同等重要，而是做加权处理：

1. 浏览权重 = 1
2. 点赞权重 = 2
3. 收藏权重 = 3

设计原因如下：

1. 浏览只能说明用户“看过”。
2. 点赞说明用户“有明显认可”。
3. 收藏说明用户“有较强兴趣，甚至希望后续继续关注”。

因此，收藏对用户兴趣画像的贡献最大，浏览贡献最小。

### 4.4 内容特征提取

系统会先把每个候选宠物、候选帖子转换成一组可计算的特征。

以宠物为例，主要包括：

1. 类别特征，如猫、狗。
2. 年龄分桶特征，如幼年、青年、成年、老年。
3. 性别与疫苗状态等结构化特征。
4. 名称、地址、详情中的文本特征。

以帖子为例，主要包括：

1. 宠物类别特征。
2. 标题关键词。
3. 摘要关键词。
4. 正文关键词。

系统在文本处理时，会先去掉 HTML 标签，再提取可用于匹配的文本 token，从而让“宠物描述”和“用户偏好”能够进入同一套相似度计算流程。

### 4.5 用户兴趣画像构建

当系统拿到用户的历史行为后，会把用户曾经浏览、点赞、收藏过的内容特征累加起来，形成用户画像。

可以把它理解为：

1. 如果用户反复浏览猫类宠物，那么“猫类”特征会被强化。
2. 如果用户多次收藏年轻、亲人、疫苗齐全的宠物，这些特征在画像中的权重会更高。
3. 如果用户经常看某类宠物经验帖，该类帖子的标题、摘要和正文关键词也会进入画像。

最终，系统得到的是一个“当前用户兴趣方向”的向量表示。

### 4.6 候选内容匹配

在得到用户兴趣画像后，系统会把候选宠物或候选帖子也表示成特征向量，然后计算：

1. 候选内容与用户画像之间是否方向一致。
2. 哪些候选内容和用户历史偏好更接近。

这里使用的核心方法是 `余弦相似度`。

余弦相似度的作用可以简单理解为：

1. 不重点比较两个向量的绝对大小。
2. 更关注两个向量的方向是否接近。
3. 方向越接近，说明候选内容越符合用户兴趣。

### 4.7 推荐结果排序

系统会根据相似度得分从高到低排序，筛出更适合当前用户的内容。

同时，系统还会做两件事：

1. 排除用户已经强互动过的内容，避免重复推荐。
2. 过滤无效内容，例如已经领养的宠物、未审核的帖子。

这样可以保证推荐结果既尽量个性化，又尽量符合前台展示规则。

### 4.8 热门内容兜底

如果当前用户行为过少，或者画像不足以形成稳定匹配结果，系统会自动回退到热门内容推荐。

兜底策略的意义在于：

1. 新用户也能看到可展示的推荐结果。
2. 行为较少时不会出现推荐列表为空。
3. 系统的推荐接口始终保持可用。

因此，当前推荐链路可以概括为：

1. 优先走个性化内容推荐。
2. 个性化结果不足时走热门内容兜底。

## 5. 为什么当前方案适合宠物领养场景

宠物领养属于低频业务，和短视频、电商不同，用户不会在极短时间内产生大量强交互数据。因此，如果完全依赖“用户与用户之间的相似性”，容易出现数据稀疏问题。

而当前方案更适合本系统，原因包括：

1. 宠物和帖子本身具有较明确的内容特征。
2. 平台行为数据虽然不是特别密集，但足以支持兴趣方向抽取。
3. 内容推荐更容易解释，便于答辩展示。
4. 热门兜底可以增强系统稳定性，避免空推荐。

本次改动后，系统在答辩展示层面具备以下特点：

1. 有业务数据总览。
